01 — The Problem

Why We Need This — Now

TokyoCleaner currently operates with 6,400+ annual orders managed efficiently through AppSheet, Zapier, and the legacy systems. But one critical gap remains: customers disappear after booking.

🔁 Repetitive Data Entry

Every new booking forces customers to re-enter their full address, room number, gate code, access instructions, photos, and 5 more fields — even if they've booked 20 times before.

10+ fields

re-entered on every single booking

👻 Zero Post-Booking Visibility

After a customer confirms their booking, they have no way to check order status, see their assigned cleaner, or view after-photos — unless they manually message on LINE or email.

0%

customer self-service post-booking

📸 Photos Lost in LINE Chat

After-photos taken by cleaners are sent to a LINE group chat and permanently lost in history. There is no structured, retrievable photo storage. Customers never see them.

100%

of after-photos unshared with customers

🗂 No Customer Data Asset

TokyoCleaner has no owned, structured customer database. Customer data is scattered across Google Sheets, and LINE chat history — with no unified profile.

~1,600+

estimated customers with no unified profile

05 — Implementation Plan

Four Phases to Full Portal

Each phase is independently valuable and can be tested before the next begins. Phase 0 requires no code and can start immediately.

🏗 Tokyo Cleaner First — Scope Philosophy This project is scoped to maximize operational benefit for TokyoCleaner's immediate workflows. Analytics, customer segmentation, and marketing automation are intentionally out of scope for all four phases. The portal is a service tool — not a data harvesting platform. Every feature decision must answer: "Does this make TokyoCleaner's operations cleaner or the customer's experience better?"
📱 Mobile-First UX for Japan Market The vast majority of consumer tech usage in Japan is mobile-based. The portal UI/UX must be strictly built mobile-first, prioritizing thumb-friendly tap targets, readable Japanese typography on narrow screens, and seamless vertical flows. Desktop is to be treated as a progressively-enhanced secondary layout.
0

Pre-Build Infrastructure — No Code Required

Start immediately · Est. 1 week · Zero customer disruption

Set up the data infrastructure before writing a single line of portal code. All new bookings made after this phase will automatically populate the portal database.

Step 1 — After-Photo Workflow Setup

  • Add "After Photos" image column to AppSheet Orders view
  • Configure upload destination: Google Drive /Orders/[order_id]/
  • Cleaners tap the field in AppSheet → take photo → auto-saves to Drive
  • Test with one cleaner. Retire LINE photo sharing after validation.

Step 2 — Zapier Dual-Write

  • Add a second action to existing Zapier zap
  • New bookings write to Supabase AND Google Sheets
  • Existing zaps remain untouched

Step 3 — Supabase Setup

  • Create Supabase project
  • Define tables: customers, properties, orders, recurring_agreements
  • Set up authentication (email magic link)

Step 4 — Recurring Agreement Manual Validation

  • Admin manually enters recurring_agreements for all current recurring customers
  • Fields: frequency, day/time, assigned cleaner, custom pricing tier
  • Pricing must be 100% human-verified before being surfaced in the portal
  • Portal acts as read-only View Layer — no customer can edit agreements

Step 5 — Weekly Reconciliation Script

  • Zapier dual-write is reliable but not immune to failures
  • Automated weekly audit compares Google Sheets ⇔ Supabase row by row
  • Any mismatches flagged for manual review before they affect the portal
  • Required before any customer behaviour analysis can be trusted
Deliverable: All new orders are dual-written to Supabase from day one. Photos go to structured Google Drive. Recurring agreements are manually validated with correct pricing. Weekly reconciliation script is active. Infrastructure is fully ready for portal development.

⚡ High-Priority Technical Bridges — Mandatory for Phase 1
1. Native Booking & Stripe Integration
By replacing Calendly with a native booking flow seamlessly integrated into the FastAPI backend, customers never leave the portal.

This allows for direct integration of the Stripe API for secure payments during checkout. Customer profiles, payment statuses, and bookings are 100% owned and handled in-house without third-party redirects.
2. Data Drift Risk — Handled in Phase 0
Zapier dual-write keeps Google Sheets and Supabase in sync, but individual zap failures can cause silent data drift — where a booking exists in one database but not the other.

The weekly reconciliation script (Phase 0, Step 5) audits both databases before any customer-facing data is shown, ensuring 100% parity. No behaviour analysis is trusted until the reconciliation baseline is established.
🔒 Privacy by Design — APPI Compliance

Property details (address, gate code, access photos) are highly sensitive. The portal never stores this data without an explicit, informed opt-in from the customer.

Express Cleaning customers
Full opt-in required. Before the Property Registry form is shown, a consent screen explains exactly what will be stored and why. Unchecked by default. Customer chooses: Save details or Fill in each time.
Recurring customers
Property details already exist in AppSheet. Portal surfaces them with full transparency: customer can view, edit, or permanently delete everything stored about their property at any time.
Consent screen wording (draft):
"To save time on future bookings, we can store your property details on your account — including your address, access instructions, and photos. You can view, update, or permanently delete this information at any time from Account Settings."
☐ Yes, save my property details    ○ No thanks, I'll fill in each time
Database records: property_data_consent (boolean) + consented_at (timestamp) stored per property for legal audit trail.
★ Existing Customer Migration — Included in This Phase

Yes — existing customers can use the portal immediately with their current email.
During Phase 0 we run a one-time migration script: all existing customer emails and past order history from Google Sheets are imported into Supabase. When an existing customer logs into the portal for the first time using the email they always used to book, they will see their complete order history already there — every past cleaning, date, status. They are then prompted to complete their property profile once. No manual re-entry of history. No confusion.

Communication to existing customers: "Log in to your new TokyoCleaner portal using [your email]. Your cleaning history is already there."

⭐ The Secret Sauce — PWA Strategy

To strictly solve the "customers disappear after booking" problem, the portal will be configured as a Progressive Web App (PWA) during Phase 1.

By bypassing the App Store entirely, TokyoCleaner gains a dedicated App Icon on the customer's phone via an in-app "Save to Homescreen" prompt. One tap instantly opens the portal in full-screen mode without browser UI — keeping TokyoCleaner permanently 1-tap away and drastically lowering the friction to re-book, view after-photos, and check schedules.

Deliverable: tokyocleaner.jp is live. Webflow is retired. Existing customers log in and see their full history immediately. New customers register property once.
